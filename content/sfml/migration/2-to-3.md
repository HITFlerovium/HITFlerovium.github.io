# 从 SFML 2 迁移到 SFML 3

欢迎来到 SFML 3！
SFML 团队付出了巨大的努力，旨在提供一个既能让现有用户感到熟悉，又带来了重大改进的库。
本文档将指导您如何将现有的 SFML 2 应用程序升级至 SFML 3。

SFML 3 最大的变化之一是将 C++ 标准提升到了 C++17，从而将 SFML 带入了现代 C++ 的世界！
这一改变不仅促成了大量的内部改进，也为优化 API 提供了新的机遇，这些我们都会在本文档中逐一探讨。

## 编译器要求

随着标准从 C++03 升级到 C++17，您可能也需要升级编译器。
在 SFML 3 发布之前的几年里，所有主流编译器均已广泛支持 C++17，因此您目前正在使用的编译器极有可能已经支持。
如果您确实需要升级，以下是最低的编译器版本要求：

| 编译器 | 版本 |
| --- | --- |
| MSVC | 16 (VS 2019) |
| GCC | 9 |
| Clang | 9 |
| AppleClang | 12 |

## CMake 目标

SFML 3 采用了现代 CMake 库目标的命名惯例，即在目标名称前添加一个命名空间。
这些命名空间明确告诉 CMake “这是一个目标”，而像 `sfml-graphics` 这样的名称，既可能是一个目标，也可能是磁盘上的一个预编译库，比如 `libsfml-graphics.so`。
消除这种歧义后，在因为（例如）忘记调用 `find_package` 而找不到某个目标时，CMake 可以提供更有用的错误信息。
此外，调用 `find_package` 时使用的组件名称现在也改为了首字母大写。

| v2 目标 | v3 目标 |
| --- | --- |
| `sfml-system` | `SFML::System` |
| `sfml-window` | `SFML::Window` |
| `sfml-graphics` | `SFML::Graphics` |
| `sfml-network` | `SFML::Network` |
| `sfml-audio` | `SFML::Audio` |
| `sfml-main` | `SFML::Main` |

| v2 组件 | v3 组件 |
| --- | --- |
| `system` | `System` |
| `window` | `Window` |
| `graphics` | `Graphics` |
| `network` | `Network` |
| `audio` | `Audio` |
| `main` | `Main` |

v2:
```cmake
find_package(SFML 2 REQUIRED COMPONENTS graphics audio network)
...
target_link_libraries(my_app PRIVATE sfml-graphics sfml-audio sfml-network)
```

v3:
```cmake
find_package(SFML 3 REQUIRED COMPONENTS Graphics Audio Network)
...
target_link_libraries(my_app PRIVATE SFML::Graphics SFML::Audio SFML::Network)
```

## Linux 依赖项

在 Linux 上使用 X11 而不是 DRM 作为后端时，`libxi-dev` 成为了新增加的必需依赖项。
这是随着支持原始鼠标输入而引入的。

## `sf::Vector2<T>` 参数

SFML 2 中一个常见的模式是使用成对的标量参数来表示诸如大小或位置之类的概念。
以 `sf::Transformable::setPosition(float, float)` 为例。
这两个参数组合起来代表了世界空间中的一个位置。

SFML 3 获取了所有类似 `(float, float)` 或 `(unsigned int, unsigned int)` 这种参数对的 API，并将它们转换为相应的 `sf::Vector2<T>` 类型，例如 `sf::Vector2f` 或 `sf::Vector2u`，从而使得接口更具表现力且更易组合。
这种过渡通常非常简单，只需用花括号将两个相邻的参数括起来以构造向量即可。

v2:
```cpp
sf::VideoMode videoMode(480, 640, 24);
sf::CircleShape circle(10);
circle.setPosition(10, 20);
sf::IntRect rect(250, 400, 50, 100);
```

v3:
```cpp
sf::VideoMode videoMode({480, 640}, 24);
sf::CircleShape circle(10);
circle.setPosition({10, 20});
sf::IntRect rect({250, 400}, {50, 100});
```

## `sf::Event`

SFML 3 在底层使用 `std::variant` 实现了一套全新的、类型安全的事件 API。
使用这套新 API 主要有两种方法。
欢迎查看新的 EventHandling 示例程序来了解这些方法的实际应用。

### `sf::Event::getIf<T>`

第一种方式围绕 `sf::Event::getIf<T>` 和 `sf::Event::is<T>` 展开。
使用 `getIf<T>` 时需要提供一个模板参数，该参数必须是事件的子类型。
事件子类型包括像 `sf::Event::Resized` 或 `sf::Event::MouseMoved` 这样的类型。
如果模板参数与当前活跃的事件子类型匹配，则返回指向该数据的指针。
如果模板参数不匹配当前的事件子类型，则返回 `nullptr`。
相比之下，如果 `T` 匹配当前活跃的事件子类型，`sf::Event::is<T>` 会直接返回 `true`。
`is<T>` 通常用于像 `sf::Event::Closed` 这样不包含数据的子类型。
具体示例如下：

```cpp
while (window.isOpen())
{
    while (const std::optional event = window.pollEvent())
    {
        if (event->is<sf::Event::Closed>())
        {
            window.close();
        }
        else if (const auto* keyPressed = event->getIf<sf::Event::KeyPressed>())
        {
            if (keyPressed->scancode == sf::Keyboard::Scancode::Escape)
                window.close();
        }
    }

    // 主循环的剩余部分
}
```

请注意获取事件的 API 发生了细微变化。
`sf::WindowBase::pollEvent` 和 `sf::WindowBase::waitEvent` 现在返回的是 `std::optional<sf::Event>`。
这两个函数*可能*会返回一个事件，但也可能不返回。
因为 C++ 允许推导模板参数，所以你可以写成 `const std::optional event` 而不是 `const std::optional<sf::Event> event`。
如果你喜欢更简洁的表达方式，`const auto event` 也是一个有效的选择。

### `sf::WindowBase::handleEvents`

处理事件的第二种方法是通过新增的 `sf::WindowBase::handleEvents` 函数。
该函数执行事件访问机制。
这意味着你可以提供将不同事件子类型作为参数的回调函数。
或者，你也可以提供一个或多个实现了 `operator()` 的对象来处理你想处理的事件子类型。
值得注意的是，你不需要为所有可能的事件子类型都提供回调。
它会根据当前活跃的事件子类型，自动调用相应的回调函数。
具体示例如下：

```cpp
const auto onClose = [&window](const sf::Event::Closed&)
{
    window.close();
};

const auto onKeyPressed = [&window](const sf::Event::KeyPressed& keyPressed)
{
    if (keyPressed.scancode == sf::Keyboard::Scancode::Escape)
        window.close();
};

while (window.isOpen())
{
    window.handleEvents(onClose, onKeyPressed);

    // 主循环的剩余部分
}
```

## 窗口样式与状态

我们新增了一个 `sf::State` 枚举，用于指定窗口的状态，即窗口是浮动还是全屏的。
这里对比了更改前后这如何影响窗口的创建过程。

```cpp
sf::RenderWindow window(sf::VideoMode::getFullscreenModes().at(0), "Title", sf::Style::Fullscreen);
```

v3:
```cpp
sf::RenderWindow window(sf::VideoMode::getFullscreenModes().at(0), "Title", sf::State::Fullscreen);
```

## `sf::Rect<T>`

`sf::Rect<T>` 已被重构，由之前的 `top`、`left`、`width` 和 `height` 四个标量值，改为了两个 `sf::Vector2<T>` 变量，分别命名为 `position` 和 `size`。
这意味着 `sf::Rect<T>::getPosition()` 和 `sf::Rect<T>::getSize()` 已经被移除，转而推荐直接访问 `position` 和 `size` 数据成员。
原本接受四个参数的构造函数也被移除，现在使用的是接受两个 `sf::Vector2<T>` 参数的构造函数。

| v2 | v3 |
| --- | --- |
| `.left` | `.position.x` |
| `.top` | `.position.y` |
| `.width` | `.size.x` |
| `.height` | `.size.y` |

v2:
```cpp
sf::FloatRect rect(10, 20, 30, 40);
sf::Vector2f position = rect.getPosition();
sf::Vector2f size = rect.getSize();
```

v3:
```cpp
sf::FloatRect rect({10, 20}, {30, 40});
sf::Vector2f position = rect.position;
sf::Vectro2f size = rect.size;
```

`sf::Rect<T>::intersects` 的两个重载方法已被合并为一个统一的函数，名为 `sf::Rect<T>::findIntersection`，它返回一个 `std::optional<Rect<T>>`。
如果矩形发生重叠，该 optional 将包含重叠区域。
否则，该 optional 将为空。

```cpp
sf::IntRect rect1({0, 0}, {200, 200});
sf::IntRect rect2({100, 100}, {200, 200});
std::optional<sf::IntRect> intersection = rect1.findIntersection(rect2);
// position={100, 100} size={100, 100}
```

## `sf::Angle`

所有的角度现在都用名为 `sf::Angle` 的强类型来表示。
这个类型提供了两个创建角度的函数：`sf::degrees(float)` 和 `sf::radians(float)`，它们分别根据给定的度数或弧度值来构造一个角度。
也提供了各种运算符（`+`、`-` 等）来执行角度的数学运算。
如果你需要以 `float` 形式访问原始的角度数值，可以使用 `sf::Angle::asDegrees()` 或 `sf::Angle::asRadians()`。

v2:
```cpp
sf::RectangleShape shape(sf::Vector2f(50, 50));
shape.setRotation(90);
std::cout << "Rotation: " << shape.getRotation() << '\n';
```

v3:
```cpp
sf::RectangleShape shape({50, 50});
shape.setRotation(sf::degrees(90));
std::cout << "Rotation: " << shape.getRotation().asDegrees() << '\n';
```

## 移除了默认构造函数

默认构造函数 `sf::Sound::Sound()`、`sf::Text::Text()` 和 `sf::Sprite::Sprite()` 已被移除。
你可以使用相对应接受资源类型参数的构造函数来代替它们。

| v2 | v3 |
| --- | --- |
| `sf::Sound::Sound()` | `sf::Sound::Sound(const sf::SoundBuffer&)` |
| `sf::Text::Text()` | `sf::Text::Text(const sf::Font&)` |
| `sf::Sprite::Sprite()` | `sf::Sprite::Sprite(const sf::Texture&)` |

现在这些类被确保一定会持有一个对其对应资源类型的引用，因此访问这些资源的函数可以返回引用而不是指针。
这些函数是：`sf::Sound::getBuffer()`、`sf::Text::getFont()` 和 `sf::Sprite::getTexture()`。

v2:
```cpp
const sf::SoundBuffer soundBuffer("sound.flac");
sf::Sound sound;
sound.setBuffer(soundBuffer);
```

v3:
```cpp
const sf::SoundBuffer soundBuffer("sound.flac");
sf::Sound sound(soundBuffer);
```

## `std::optional` 的使用

SFML 3 大量使用了 `std::optional` 来表示某个函数可能无法返回值的情况。
其中一些用法已经提及过，例如 `sf::WindowBase::pollEvent`。
以下是 SFML 3 中更多使用了 `std::optional` 的地方：

*   `sf::IpAddress` 使用 `std::optional` 来表达从字符串解析地址时，可能无法生成一个可用的 IP 地址。
*   `sf::Image::saveToMemory` 返回 `std::optional` 是因为 `sf::Image` 可能为空，或者底层的实现可能会失败。
*   `sf::SoundFileReader::open` 返回 `std::optional` 是因为被打开的流可能无效。
*   `sf::Music::onLoop` 和 `sf::SoundStream::onLoop` 返回 `std::optional`，因为如果对象未处于循环状态，则无数据可返回。
*   `sf::InputStream` 在多个地方使用了 `std::optional`。
与返回 `-1` 来表示错误不同，它可以返回 `std::nullopt`。

LearnCpp.com 是一个学习 `std::optional` 使用方法的绝佳去处。
你可以在[这里](https://www.learncpp.com/cpp-tutorial/stdoptional/)阅读更多相关内容。

## 线程原语

`sf::Lock`、`sf::Mutex`、`sf::Thread`、`sf::ThreadLocal` 以及 `sf::ThreadLocalPtr` 均被移除，并由标准库中相应的组件取代。
标准库在线程、锁和互斥量等线程原语方面提供了多样的选择。

| v2 | v3 |
| --- | --- |
| `sf::Lock` | `std::lock_guard` 或 `std::unique_lock` |
| `sf::Mutex` | `std::mutex` 或 `std::recursive_mutex` |
| `sf::Thread` | `std::thread` 或 `std::jthread` (要求 C++20) |
| `sf::ThreadLocal` | `thread_local` |
| `sf::ThreadLocalPtr` | `thread_local` |

## 音频采样与声道映射

SFML 3 引入了声道映射的概念，它定义了音频采样帧中的位置到具体声道的映射关系。
例如，如果一个音频帧包含对应 5.1 音响系统的六个不同采样，那么声道映射将定义这每一个采样各自对应的是哪个扬声器声道。

SFML 2 始终假定采用了 OpenAL 所指定的顺序。

```cpp
auto samples = std::vector<std::int16_t>();
// ...

auto channelMap = std::vector<sf::SoundChannel>{
    sf::SoundChannel::FrontLeft,
    sf::SoundChannel::FrontCenter,
    sf::SoundChannel::FrontRight,
    sf::SoundChannel::BackRight,
    sf::SoundChannel::BackLeft,
    sf::SoundChannel::LowFrequencyEffects
};
auto soundBuffer = sf::SoundBuffer(samples.data(), samples.size(), channelMap.size(), 44100, channelMap);
auto sound = sf::Sound(soundBuffer);
```

这是一个破坏性变更，影响了以下 API：

*   `bool sf::SoundBuffer::loadFromSamples(...)`
*   `bool sf::SoundBuffer::update(...)`
*   `void sf::SoundStream::initialize(...)`
*   `bool sf::OutputSoundFile::openFromFile(...)`
*   `bool sf::SoundFileWriter::open(...)`

| `sf::PrimitiveType::TrianglesFan` | `sf::PrimitiveType::TriangleFan` |
| `sf::PrimitiveType::Quads` | 见说明 2 |
| `sf::Keyboard::BackSlash` | `sf::Keyboard::Key::Backslash` |
| `sf::Keyboard::BackSpace` | `sf::Keyboard::Key::Backspace` |
| `sf::Keyboard::Dash` | `sf::Keyboard::Key::Dash` |

## 抗锯齿重命名

SFML 3 将所有相关 API 中的 `aliasing` 的首字母 `A` 进行了大写处理。

*   `sf::RenderTexture::getMaximumAntialiasingLevel()` 变为了 `sf::RenderTexture::getMaximumAntiAliasingLevel()`
*   `sf::ContextSettings::antialiasingLevel` 变为了 `sf::ContextSettings::antiAliasingLevel`

## RenderStates 的 CoordinateType

枚举 `sf::CoordinateType` 从 `sf::Texture` 中被移出，并拥有了自己专属的强类型枚举类。

`sf::RenderStates` 类新增了一个名为 `sf::CoordinateType` 的成员，用于控制纹理坐标的解析方式。
默认情况下 SFML 使用 `sf::CoordinateType::Pixels`，而 OpenGL 的默认值则是 `sf::CoordinateType::Normalized`。
在 `sf::RenderStates` 中搭配使用 `sf::CoordinateType::Normalized`，使得规范化的纹理能够应用在 `sf::VertexArray` 以及 `sf::VertexBuffer` 上。

同时，`sf::RenderStates` 的构造函数也发生了改变。

v2:
```cpp
auto renderStates = sf::RenderStates(sf::BlendAlpha,
                                     transform,
                                     texture,
                                     nullptr);
```

v3:
```cpp
auto renderStates = sf::RenderStates(sf::BlendMode::BlendAlpha,
                                     transform,
                                     sf::CoordinateTye::Pixels,
                                     texture,
                                     nullptr);
```

## 其他微小变更

SFML 3 还有各种值得一提的微小变更。

*   更改了 `sf::Text` 构造函数的参数顺序，使得被提供的字体参数始终位于第一位。
*   恢复了 CMake 的 `BUILD_SHARED_LIBS` 的默认行为，这意味着 SFML 现在默认编译为静态库。
*   更改了 `sf::String` 接口以使用 `std::u16string` 和 `std::u32string`。
*   移除了 `sf::ContextSettings` 的构造函数，推荐使用聚合初始化。
*   移除了 `sf::View::reset`，现在推荐直接通过赋值一个新的 `sf::View` 对象来实现重置。
*   移除了 `sf::Vertex` 构造函数，推荐使用聚合初始化。
*   将枚举成员 `sf::Mouse::Button::XButton1` 和 `sf::Mouse::Button::XButton2` 重命名为了 `sf::Mouse::Button::Extra1` 和 `sf::Mouse::Button::Extra2`。
*   移除了 NonCopyable.hpp 头文件，推荐直接利用语言内建特性来禁用拷贝运算符。
*   将以下这些类转换为了命名空间：`sf::Clipboard`、`sf::Keyboard`、`sf::Joystick`、`sf::Listener`、`sf::Mouse`、`sf::Sensor`、`sf::Touch`、`sf::Vulkan`。
*   由于 miniaudio 会自动使内部处理速率匹配底层后端的频率，因此移除了 `sf::SoundStream::setProcessingInterval`。
