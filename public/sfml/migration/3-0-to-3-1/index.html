SFML 3.0 到 3.1 迁移指南

July 10, 2026 · 6 min · Flerovium

从 SFML 3.0 迁移到 SFML 3.1#

欢迎来到 SFML 3.1！ 本次发布带来了在文本布局、网络、音频设备管理等方面的重大改进。

依赖项#

我们引入了一系列新的依赖项以支持所有改进：

  • FetchContent / 系统依赖
    • HarfBuzz 用于复杂的文本布局
    • SheenBidi 用于正确的双向文本渲染
    • Mbed TLS 提供 TLS 支持
    • libssh2 提供 SFTP 支持
  • Header-only (仅头文件)
    • cpp-unicodelib 提供完善的 Unicode 支持
    • wepoll 用于改进 SocketSelector
    • qoi 支持 QOI 图像格式

废弃接口#

已废弃的函数仍然可用且功能完整，不过您可能会收到编译器警告。 建议迁移到下面描述的替代方案，因为它们通常能提供显著的改进。

字体字距调节#

使用 std::uint32_t 表示 Unicode 码位的 sf::Font::getKerning 重载现已废弃。 请改用接受 char32_t 的重载版本。 两者的行为完全一致——仅仅是参数类型发生了改变。

v3.0:

float kerning = font.getKerning(static_cast<std::uint32_t>('A'),
                                static_cast<std::uint32_t>('V'),
                                30);

v3.1:

float kerning = font.getKerning(U'A', U'V', 30);

文本中的字符位置#

sf::Text::findCharacterPos(std::size_t index) 已被废弃，替代方案为 getShapedGlyphs()。

旧函数返回单个字符在全局空间中的位置。 新函数返回文本整形期间生成的完整 ShapedGlyph 对象列表，除了底层的 sf::Glyph 数据外，还允许您访问每个字形的局部位置、簇 ID、文本方向以及基线。

struct ShapedGlyph
{
    Glyph         glyph;         // 字形数据
    Vector2f      position;      // 在文本内的局部位置
    std::uint32_t cluster;       // 簇 ID（多个字形可能会共享同一个簇）
    TextDirection textDirection; // 周围文本序列的方向
    float         baseline;      // 该字形所在行的基线 Y 位置
    std::size_t   vertexOffset;  // 顶点缓冲区的起始偏移量
    std::size_t   vertexCount;   // 该字形的顶点数量
};

要进行迁移，请遍历这些整形后的字形并使用其 position 字段。

注意：局部坐标与全局坐标

请注意，getShapedGlyphs() 返回的是局部坐标，而 findCharacterPos() 返回的是全局坐标。 如果您需要全局坐标，请自行应用文本的变换。

v3.0:

// 查找第 5 个字符的位置（全局空间）
sf::Vector2f pos = text.findCharacterPos(4);

v3.1:

// 查找属于第 5 个字符的字形的位置（局部空间）
const auto& glyphs = text.getShapedGlyphs();
if (glyphs.size() > 4)
{
    sf::Vector2f localPos = glyphs[4].position;
    // 如果需要，可以转换为全局空间：
    sf::Vector2f globalPos = text.getTransform().transformPoint(localPos);
}

多个 Unicode 码位可以组成一个单一的字位簇（例如：一个基本字符加上一个组合重音符号）。 当发生这种情况时，字形列表中的几个连续条目将共享同一个 cluster 值。 如果您正在实现一个文本光标，请将同一簇中的所有字形视为一个单独的单元。

实时触摸输入#

sf::Touch::isDown(unsigned int finger) 以及 sf::Touch::getPosition(unsigned int finger) 的两个重载均已废弃。 在 Android 平台上，当查询一个当前未按下的手指时，这些函数可能会导致崩溃；而事件驱动的方法则始终是安全的。

请改用相关触摸事件的 position 成员：

废弃的函数 替代方案
sf::Touch::isDown sf::Event::TouchBegan / sf::Event::TouchEnded
sf::Touch::getPosition sf::Event::TouchBegan、sf::Event::TouchMoved 或 sf::Event::TouchEnded 的 position 字段

v3.0:

if (sf::Touch::isDown(0))
{
    sf::Vector2i pos = sf::Touch::getPosition(0);
    // 使用 pos
}

v3.1:

// 在您的事件循环中：
window.handleEvents([&](const sf::Event::TouchBegan& touch)
{
    if (touch.finger == 0)
    {
        sf::Vector2i pos = touch.position;
        // 使用 pos
    }
},
[&](const sf::Event::TouchMoved& touch)
{
    if (touch.finger == 0)
    {
        sf::Vector2i pos = touch.position;
        // 使用 pos
    }
});

FTP 客户端#

sf::Ftp 已被废弃，替代方案为全新的 sf::Sftp 类（参见下文的新特性）。 FTP 以明文形式传输凭据和数据，这使得 SFTP 成为任何新代码的强烈推荐选择。

主机名解析#

sf::IpAddress::resolve(std::string_view address) 已被废弃。 它只能解析为 IPv4 地址，并且缺乏对 DNS 服务器或超时的控制。

请改用 sf::Dns::resolve()，它同时支持 IPv4 和 IPv6，并为您提供对查询的完全控制（参见下文的新特性）。

v3.0:

std::optional<sf::IpAddress> address = sf::IpAddress::resolve("www.sfml-dev.org");
if (address)
{
    // 使用 *address
}

v3.1:

std::optional<std::vector<sf::IpAddress>> addresses = sf::Dns::resolve("www.sfml-dev.org");
if (addresses && !addresses->empty())
{
    sf::IpAddress address = addresses->front();
    // 使用 address
}

新特性#

运行时版本查询#

sf::version() 返回一个描述在运行时加载的 SFML 库的 const sf::Version&。 当您的应用程序是动态链接，并且您想要验证运行时库是否与您编译时使用的头文件匹配时，这非常有用。

const sf::Version& v = sf::version();
// v.major, v.minor, v.patch - 数字组件
// v.isRelease               - 对于开发构建为 false
// v.string                  - 例如 "3.1.0" 或 "3.1.0-dev"

if (v.major != SFML_VERSION_MAJOR || v.minor != SFML_VERSION_MINOR)
{
    // 编译时和运行时的 SFML 版本不匹配
}

改进的文本布局与整形#

SFML 3.1 引入了由 HarfBuzz、SheenBidi 和 cpp-unicodelib（Unicode 17.0）驱动的、支持 Unicode 的文本整形。 这意味着现在的字形是根据其对应书写系统的规则来定位的——包括连字替换、标记附加和双向重排——而不是像以前那样仅仅通过简单的字形表查找来逐个字符放置。

对于绝大多数纯 ASCII 或拉丁文文本，视觉效果与以前完全一致。 当涉及到复杂书写系统（阿拉伯文、希伯来文、天城文、中日韩文字等）、连字、组合标记以及从右到左（RTL）或垂直文本时，这些改进就会显现出来。

从右到左与双向文本#

SFML 3.1 可以自动检测并处理从右到左（RTL）的书写系统（如阿拉伯文和希伯来文），以及在同一字符串中混合了 LTR 和 RTL 序列的双向（BiDi）文本。 无需任何配置：只需分配一个 Unicode 字符串，SFML 就会自动应用 Unicode 双向算法。

// 纯 RTL - 阿拉伯文或希伯来文会自动从右向左渲染
sf::Text arabicText(font, U"\u0645\u0631\u062d\u0628\u0627", 30); // "مرحبا"

// 混合 BiDi - 同一个字符串中包含 LTR 的英文和 RTL 的希伯来文
sf::Text bidiText(font, U"Hello \u05e9\u05dc\u05d5\u05dd World", 30);

getShapedGlyphs() 返回的列表中的每个 ShapedGlyph 都带有一个 textDirection 字段，该字段反映了该字形所属文本序列的解析方向：

for (const sf::Text::ShapedGlyph& g : text.getShapedGlyphs())
{
    if (g.textDirection == sf::Text::TextDirection::RightToLeft)
    {
        // 该字形属于一个从右到左的文本序列
    }
}

行对齐#

默认情况下，每一行都会根据其主导书写系统的方向进行对齐：LTR 的行左对齐，RTL 的行右对齐。 您可以通过 setLineAlignment() 覆盖此行为：

// 默认：感知书写系统方向的对齐方式
text.setLineAlignment(sf::Text::LineAlignment::Default);

// 强制所有行从左侧开始，无论书写系统方向如何
text.setLineAlignment(sf::Text::LineAlignment::Left);

// 居中所有行
text.setLineAlignment(sf::Text::LineAlignment::Center);

// 强制所有行在右侧结束
text.setLineAlignment(sf::Text::LineAlignment::Right);

这对于用户界面布局非常有用，因为您希望无论内容语言是什么，都能保持一致的视觉对齐。

垂直文本方向#

setTextOrientation() 将布局轴从水平切换为垂直。 这主要针对原生包含了垂直度量字体的东亚书写系统（中文、日文、韩文）。

// 水平布局（默认）
text.setTextOrientation(sf::Text::TextOrientation::Default);

// 垂直，从上到下
text.setTextOrientation(sf::Text::TextOrientation::TopToBottom);

// 垂直，从下到上
text.setTextOrientation(sf::Text::TextOrientation::BottomToTop);

注意

当选择了垂直方向时，步进宽度和基线位置将沿着垂直轴解释。 不提供原生垂直度量的字体将使用模拟值；对于所有书写系统来说，结果可能并不总是视觉上最理想的。

簇分组#

文本整形可以将多个输入的码位合并为一个单一的渲染字形（连字），或者将一个单一的码位分解为多个字形（基础字符 + 标记）。 簇分组控制着这种多对多映射是如何通过 getShapedGlyphs() 暴露出来的，当您需要将一个字形关联回输入字符串中的位置时（例如，放置一个文本光标），这一点就很重要。

// 字符簇（默认）- 适用于大多数光标定位用例
text.setClusterGrouping(sf::Text::ClusterGrouping::Character);

// 字位簇 - 将组合序列分组为一个簇
// （例如：一个基础字符 + 组合重音符号 = 一个簇）
text.setClusterGrouping(sf::Text::ClusterGrouping::Grapheme);

// 不分组 - 每个整形后的字形都会获得自己的簇值
text.setClusterGrouping(sf::Text::ClusterGrouping::None);

每个 ShapedGlyph 的 cluster 字段映射回输入字符串中的一个索引。 当连续的多个字形构成一个不可分割的单元（例如，跨越两个输入字符的连字）时，它们可能会共享同一个 cluster 值。 请将所有具有相同簇值的字形视为一个单独的光标停靠点。

// 遍历整形后的字形并收集每个唯一簇的 x 位置
// （适用于为 LTR 文本构建光标位置表）
std::map<std::uint32_t, float> cursorPositions;
for (const sf::Text::ShapedGlyph& g : text.getShapedGlyphs())
{
    cursorPositions.try_emplace(g.cluster, g.position.x);
}

使用 GlyphPreProcessor 进行逐字形样式设置#

setGlyphPreProcessor() 会安装一个回调函数，在几何体生成期间，每个字形都会触发该回调。 它允许您逐个字形地覆盖 style、fillColor、outlineColor 和 outlineThickness，从而无需管理多个 sf::Text 对象即可实现诸如语法高亮等效果。

该回调接收整形后的字形（包括其 cluster 索引）以及逐字形渲染属性的可变引用：

// 使用输入字符串中的码位索引范围为特定单词设置不同的颜色
// 对于输入 "I like flowers, muffins and waffles."
// "muffins" 跨越码位索引 16–22
text.setGlyphPreProcessor(
    [](const sf::Text::ShapedGlyph& glyph,
       std::uint32_t& /*style*/,
       sf::Color&     fillColor,
       sf::Color&     /*outlineColor*/,
       float&         /*outlineThickness*/)
    {
        if (glyph.cluster >= 16 && glyph.cluster <= 22)
        {
            fillColor = sf::Color::Red;
        }
    });

因为文本整形可能会将相邻的码位合并为连字（例如 fl → ﬂ），所以安全的逐词着色应该使用簇索引范围，而不是字形索引范围。 只要字形的 cluster 落在该单词的码位范围内，它就属于该单词，即使它是跨越了单词边界的连字。

该回调按几何体生成的顺序被调用，即视觉顺序（而不是逻辑顺序）。 对于 RTL（从右到左）文本，这意味着随着回调的进行，簇值会递减。

警告

请勿在回调内部调用 getLocalBounds()；此时几何体仍在构建中。

TLS / HTTPS 支持#

sf::Http 现在支持 HTTPS。 将一个 https:// 的 URL 传递给 sf::Http::setHost()，SFML 将自动建立 TLS 加密连接。

sf::Http http;
http.setHost("https://www.sfml-dev.org");

sf::Http::Request request("/");
sf::Http::Response response = http.sendRequest(request);

如需禁用服务器证书验证（不推荐用于生产环境）：

sf::Http::Response response = http.sendRequest(request, sf::Time::Zero, /* verifyServer */ false);

DNS API#

全新的 sf::Dns 命名空间提供了完整的 DNS 查询支持，取代了已废弃的 sf::IpAddress::resolve。

解析主机名（A / AAAA 记录）：

#include <SFML/Network/Dns.hpp>

std::optional<std::vector<sf::IpAddress>> addresses = sf::Dns::resolve("www.sfml-dev.org");
if (addresses)
{
    for (const sf::IpAddress& addr : *addresses)
    {
        if (addr.isV4())
            std::cout << "IPv4: " << addr.toString() << '\n';
        else
            std::cout << "IPv6: " << addr.toString() << '\n';
    }
}

查询其他记录类型：

// 域名服务器记录
std::vector<sf::String> ns = sf::Dns::queryNs("sfml-dev.org");

// 邮件交换记录
std::vector<sf::Dns::MxRecord> mx = sf::Dns::queryMx("sfml-dev.org");
// mx[i].exchange - 邮件服务器的主机名
// mx[i].preference - 值越低优先级越高

// 服务记录
std::vector<sf::Dns::SrvRecord> srv = sf::Dns::querySrv("_irc._tcp.sfml-dev.org");
// srv[i].target, srv[i].port, srv[i].weight, srv[i].priority

// 文本记录
std::vector<std::vector<sf::String>> txt = sf::Dns::queryTxt("sfml-dev.org");

获取本机的公网 IP 地址：

// IPv4
std::optional<sf::IpAddress> pubV4 = sf::Dns::getPublicAddress(sf::seconds(5));

// IPv6
std::optional<sf::IpAddress> pubV6 = sf::Dns::getPublicAddress(sf::seconds(5), sf::IpAddress::Type::IpV6);

所有 sf::Dns 函数都接受可选的自定义 DNS 服务器列表以及可选的超时参数。

SFTP 客户端#

全新的 sf::Sftp 类提供了一个完整的 SFTP（SSH 文件传输协议）客户端。 它同时支持用户名/密码和公钥身份验证，取代了已废弃的 sf::Ftp。

连接并登录：

#include <SFML/Network/Sftp.hpp>

sf::Sftp sftp;

auto addresses = sf::Dns::resolve("sftp.example.com");
sf::Sftp::Result result = sftp.connect(addresses->front());
if (result.getValue() != sf::Sftp::Result::Value::Ok)
{
    return; // 连接失败
}

// 密码认证
result = sftp.login("alice", "hunter2");

// 或者公钥认证
result = sftp.login("alice",
                    std::string_view{publicKey},
                    std::string_view{privateKey},
                    "passphrase");

文件操作：

// 列出目录
sf::Sftp::ListingResult listing = sftp.getDirectoryListing("/home/alice");
for (const auto& entry : listing.getListing())
{
    std::cout << entry.name << '\n';
}

// 通过回调下载文件
std::vector<std::uint8_t> buffer;
sftp.download("/home/alice/data.bin",
    [&buffer](const void* data, std::size_t size)
    {
        const auto* bytes = static_cast<const std::uint8_t*>(data);
        buffer.insert(buffer.end(), bytes, bytes + size);
        return true; // 返回 false 即可中止
    });

// 通过回调上传文件
std::string content = "hello, world";
std::size_t offset = 0;
sftp.upload("/home/alice/hello.txt",
    [&content, &offset](void* data, std::size_t& size) -> bool
    {
        std::size_t remaining = content.size() - offset;
        if (remaining == 0)
            return false; // 提示数据结束
        size = std::min(size, remaining);
        std::memcpy(data, content.data() + offset, size);
        offset += size;
        return true;
    });

// 其他操作
sftp.createDirectory("/home/alice/newdir");
sftp.deleteFile("/home/alice/old.txt");
sftp.rename("/home/alice/a.txt", "/home/alice/b.txt");
sftp.disconnect();

sf::SocketSelector 回调 API#

sf::SocketSelector::add() 现在接受一个可选的就绪回调，支持可扩展的事件驱动设计，从而避免了每次调用 wait() 后都必须遍历所有套接字。 传统的 isReady() 轮询方法仍能正常使用。

传统方法（依然有效）：

selector.add(listener);
selector.add(client);

while (selector.wait())
{
    if (selector.isReady(listener))
    {
        // 接受新连接
    }
    if (selector.isReady(client))
    {
        // 从客户端接收数据
    }
}

全新的回调方法：

selector.add(listener, sf::SocketSelector::Receive, [&](sf::SocketSelector::ReadinessType)
{
    // 接受新连接
    auto client = std::make_unique<sf::TcpSocket>();
    listener.accept(*client);
    selector.add(*client, sf::SocketSelector::Receive,
        [c = client.get()](sf::SocketSelector::ReadinessType)
        {
            // 从客户端接收数据
        });
    clients.push_back(std::move(client));
});

while (selector.wait())
{
    selector.dispatchReadyCallbacks();
}

在管理大量并发连接时，回调方法只需要检查就绪的套接字，而不像轮询方法那样需要遍历所有套接字。

sf::Http::setHost 返回值#

sf::Http::setHost 现在返回一个 bool 值，以指示提供的主机名是否被解析以及是否有效。

v3.0:

http.setHost("https://www.sfml-dev.org"); // 此前返回值是 void

v3.1:

if (!http.setHost("https://www.sfml-dev.org"))
{
    // 主机名无法解析
}

sf::Http::sendRequest 现为 const#

sf::Http::sendRequest 现在是一个 const 成员函数，这允许从 const sf::Http 对象或引用中发送 HTTP 请求。

音频播放设备采样率#

sf::PlaybackDevice::getDeviceSampleRate() 能够返回当前活动音频输出设备的原生采样率。 此功能可用于配置音频源以匹配设备的首选采样率，避免不必要的重采样。

if (auto rate = sf::PlaybackDevice::getDeviceSampleRate())
{
    std::cout << "Device sample rate: " << *rate << " Hz\n";
}
else
{
    std::cout << "No playback device available\n";
}

QOI 图像格式#

SFML 3.1 增加了对 QOI 格式的支持。 你可以像使用其他受支持的格式一样加载和保存 QOI 文件。 加载时的格式检测是根据 .qoi 文件后缀名或文件的魔数自动完成的。

// 加载
sf::Image image;
if (!image.loadFromFile("texture.qoi"))
{
    return -1;
}

// 保存到文件
image.saveToFile("output.qoi");

// 保存到内存
std::optional<std::vector<std::uint8_t>> data = image.saveToMemory("qoi");

sf::String 支持传入 std::string_view#

sf::String 现在提供了接受 std::string_view、std::wstring_view 以及 std::u32string_view 的构造函数。 这消除了仅仅为了将其传递给 SFML 而特意构造一个 std::string 的需要。

std::string_view sv = "hello, world";
sf::String s(sv); // 不再需要使用临时的 std::string

sf::Vector3u 类型别名#

新增了 sf::Vector3u 作为 sf::Vector3<unsigned int> 的类型别名，使其与现有的 sf::Vector2u 命名约定保持一致。

sf::Vector3u size{800u, 600u, 32u};
